前端权限控制

前端权限控制

权限控制

  当系统适用于不同用户，而不同用户所使用到的系统功能存在差异的时候，权限控制就显得比较重要。此外，考虑安全性和性能方面的需求，权限控制也是不可缺少的。

前端权限控制的作用

  从技术层面来讲，权限控制可以只在后端实现。但前端的权限控制也有不可或缺的作用。

  1. 隔离操作，提升突破权限的门槛；
  2. 提升系统性能，避免和过滤越权请求，减轻服务端压力；
  3. 提升用户体验；

    前端实现权限控制

    权限划分

      权限控制的第一步是权限划分，首先要知道用户具有哪些权限才能进行管理，具体可参考RBAC模型。

    权限数据

      对前端而言，在用户登录后第一步就是获取权限数据，不同系统的权限数据可能存在差异，但最终指向基本都是路由权限和资源权限。

路由权限： 用户可访问的路由集合，以此作为设置前端路由和生成导航菜单的依据；
资源权限： 用户可访问的资源集合，“资源” 概念来自 RESTful 架构，通俗意义即为API；

权限分配

  由于权限分配可能涉及到很多资源集合，单个添加尤为繁琐，因此很多系统会通过角色来分配权限，授权时只需要对用户赋予特定的角色即可。

权限控制

路由控制： 路由控制的思路有两种，一种是初始挂载全部路由，每次路由跳转前做校验；另一种是只挂载用户有权限访问的路由，从源头控制。前者每一次路由跳转都要做一次校验，比较浪费资源，对于用户无权访问的路由，最好的实现方式就是不挂载。
资源控制： 资源控制原理是对请求作拦截，在请求拦截器中对请求的url和资源权限数据作比对，再判断是否进行拦截。根据url的不同可考虑采用模式匹配，前后端最好协商一致以便处理。

常用登录鉴权方式

常用登录鉴权方式

Basic鉴权

  HTTP/1.0定义的认证方式，一般用于内部局域网系统FTP登录，使用较少。
鉴权流程：
  1. 客户端向服务端发送请求，服务端检查请求头中是否包含Authorization字段，若没有则返回401状态码，并在响应头附加一个字段WWW-Authenticate，包含Basic（认证方式）与realm（安全域）两个值。
  2. 客户端接收到401状态码后，为了通过认证，需要将用户名及密码发送给服务器，发送的形式为用户名和密码组成的字符串，中间以冒号连接，再经过Base64编码处理。（此过程一般浏览器会进行处理，触发一个自带的弹窗，提示输入用户名和密码进行验证）。
  3. 服务端接收到包含Authorization请求头字段的请求，对认证的信息进行校验，通过验证后，返回对应的资源响应。

WWW-Authenticate: Basic realm=""     
Authorization: Basic Base64(username:password)

Digest鉴权

  HTTP/1.1新增的认证方式，弥补了Basic认证直接发送明文密码的缺点，改用质询响应方式替代实现。
鉴权流程：
  1. 客户端向服务端发送请求，服务端检查请求头中是否包含Authorization字段，若没有则返回401状态码，并在响应头附加一个字段WWW-Authenticate，此字段必须包含realm和nonce两个值，客户端依赖这两个值进行认证。此外，有一些附加值可用于限制。
  2. 客户端接收到401状态码后，返回包含认证必须的首部字段Authorization信息。首部字段Authorization内必须包含 username、realm、nonce、uri 和response的字段信息。其中，realm 和nonce为之前从服务器接收到的响应中的值。
  3. 服务端接收到包含Authorization请求头字段的请求，对认证的信息进行校验，通过验证后，返回对应的资源响应。

The WWW-Authentaicate Response Header：
realm: 域值，是一个可选参数，应至少包含主机名，代表一个授权范围。
nonce: 服务端产生的随机数，用于增加验证信息的复杂性。
qop-options: 质量保护，可选参数，默认值为auth，用于标识认证的程度，auth-int包含了完整性校验。
stale: nonce过期标志，不用于初始校验。
algorithm: 摘要算法。
The Authorization Request Header
username: 用户名。
digest-uri: 请求的URI。
qop: qop-options中的可选值，表示选择的质量保护类型。
cnonce: 由客户端产生的随机数，如果指定了qop值，则必须设定此值。
response: 32位16进制字符串，包含加密后的用户信息。
nc: 8位16进制数，标记请求的次数，结合nonce预防重放攻击。

session-cookie鉴权

  利用服务器端的session和浏览器端的cookie来实现认证。
鉴权流程：
  1. 服务端在接受客户端首次访问时在服务端创建session，然后将session保存在内存或redis中，给session生成一个唯一的标识字符串sessionId，将其加入响应头。
  2. 客户端收到响应后进行解析，将sessionId保存在cookie中，在下次请求时请求头中会带上该域名下的cookie信息。
  3. 服务端解析请求头cookie中的sessionId，然后根据这个sessionId去匹配保存的客户端session，判断是否合法。

Token鉴权

鉴权流程：
  1. 客户端使用用户名和密码发起登录请求。
  2. 服务端收到请求，验证用户名与密码。
  3. 验证成功后，服务端签发一个Token，返回给客户端。
  4. 客户端收到Token后进行存储，可放在cookie或者Local Storage中。
  5. 客户端每次向服务端请求资源的时候带着服务端签发的Token。
  6. 服务端收到请求，然后去验证客户端请求中携带的Token，验证成功则向客户端返回请求的数据。

oAuth鉴权

  OAuth（开放授权）是一个开放标准，允许用户授权第三方网站访问它们存储在其它服务提供者上的信息，而不需要将用户名和密码提供给第三方网站或分享它们数据的所有内容，为了保护用户数据的安全和隐私，第三方网站访问用户数据前都需要显式的向用户征求授权。
鉴权流程：
  1. 向用户请求授权，通过登录网站放置的第三方登录入口点击登录，由第三方授权服务引导进入第三方登录授权页面。
  2. 当用户点击授权并登录后，授权服务器会返回一个用户凭证。
  3. 第三方应用后台通过第二步的凭证向授权服务器请求Access Token。
  4. 授权服务器同意授权后，返回一个资源访问的凭证Access Token。
  5. 第三方应用通过第四步返回的Access Token登录网站请求资源。
  6. 登录网站验证通过后，返回对应的资源。

为什么通过登录后每次请求仍需要携带鉴权信息？

  1. HTTP协议是无状态的，服务器不会保存会话信息，因此需要一种机制来判断用户是否登录过，对每一次HTTP请求进行验证。
  2. 防止用户跳过登录页面，直接访问内部页面。

常见的攻击及防御

发表于 2020-02-10 | 分类于 JavaScript

常见的攻击及防御

XSS（Cross Site Scripting）跨站脚本攻击

攻击原理： 利用站点内的信任用户，向页面中注入可执行的脚本。
攻击类型： 反射型和存储型。
防御方法：

  1. 对提交的数据进行验证，过滤“<”，“>”，“ ‘ ”，“ “ ”等字符，排查javascript等相关可执行的代码。
  2. 对输出到页面的数据进行encode，避免直接显示html()函数显示内容。
  3. 避免直接在cookie中存放敏感数据。
  4. 使用HTTP only的cookie，禁止通过js获取cookie。
  5. 浏览器自带防御机制，在http头中打开X-XSS-Protection，默认值为1，打开状态。

CSRF（Cross Site Request Forgy）跨站请求伪造

攻击原理： 通过伪装来自受信任用户的请求进行攻击，攻击建立在浏览器与服务器的会话过程中，通过欺骗用户发送请求达到攻击目的。
漏洞检测： 通过抓包工具抓取数据包，去掉Referer字段后重新提交，若提交有效，则存在CSRF漏洞。（CSRF Tester, CSRF Request Builder等）
防御方法：

  1. 验证HTTP Referer字段。根据HTTP协议，在HTTP头中有一个字段Referer，它记录了该HTTP请求的来源地址。在通常情况下，访问一个安全受限页面的请求来自于同一个网站，因此，只要对每一个请求验证其Referer值即可判断该请求是否合法。
  2. 在HTTP请求中以参数的形式加入一个随机产生的token,服务端通过校验请求中的token判断请求是否非法。
  3. 在HTTP头中添加自定义属性，与方法2类似，也是通过添加Token进行验证，但是把Token放到了HTTP头的自定义属性中，避免添加参数的繁琐性。
  4. 设置same-site属性，限制第三方请求携带本网站的cookie。
  5. 设置验证码，发起请求后，只有输入正确的验证码才会进行相关操作。

SQL注入

Ddos（Distributed Denial of Service）分布式拒绝服务攻击

攻击原理： 利用网络协议和操作系统的一些缺陷，采用欺骗和伪装的策略来进行网络攻击，使网站服务器充斥大量要求回复的信息，消耗网络带宽或系统资源，导致网络或系统不胜负荷以至于瘫痪而停止提供正常的网络服务。

前端模块化

什么是模块化？

  将一个复杂的程序分解成多个独立的模块的代码组织方式，模块内部的数据与方法是私有的, 只是通过向外部暴露一些接口来实现与外部其它模块之间的通信。

为什么要模块化？

  解决程序之间的依赖关系以及命名冲突，更高的复用性和可维护性。

模块化发展历程

对象封装

  通过将数据和方法封装到一个全局对象上实现。
优点：减少了全局变量，解决了命名冲突；
缺点：数据不安全，外部可以直接修改模块内部的数据；

IIFE模式

  将数据和行为封装到一个函数内部, 通过给window添加属性来向外暴露接口。
优点：数据是私有的，外部只能通过暴露的方法操作；
缺点：模块之间的依赖性无法体现；

引入多个js文件

优点：相比使用一个js文件，实现了最简单的模块化思想；
缺点：污染全局作用域，依赖关系不明显，不利于维护；

模块化规范

第一阶段：CommonJS规范

  CommonJS是一个JavaScript模块化的规范，最初用于服务器端的node.js中，前端的webpack对CommonJS也是原生支持的，核心思想是通过require方法来同步加载所要依赖的其他模块，然后通过exports或者module.exports来导出需要暴露的接口，在运行时加载所有模块并生成一个对象；
优点：CommonJS规范在服务器端率先完成了JavaScript的模块化，解决了依赖、全局变量污染的问题；
缺点：CommonJS是同步加载模块的，在服务器端，文件都是保存在硬盘上，所以同步加载没有问题，但对于浏览器端则不适用；

第二阶段：AMD规范

  AMD规范是异步加载模块，实现的js库为require.js，允许指定回调函数，AMD中通过define来定义一个模块，然后使用require来加载一个模块，并且require还支持CommonJS的模块导出方式；
优点：适合在浏览器环境中异步加载模块，可以并行加载多个模块；
缺点：提高了开发成本，并且不能按需加载，而是必须提前加载所有的依赖；

第三阶段：CMD规范

  实现的js库为sea.js，与require.js非常类似，一个js文件就是一个模块，但CMD的加载方式是按需加载，而不是在模块开始就加载所有的依赖；
优点：实现了浏览器端的模块化加载，可以按需加载，依赖就近；
缺点：依赖SPM打包，模块的加载逻辑偏重；

第四阶段：ES6模块化

  ES6模块的设计思想是尽量的静态化，在编译时确定模块的依赖关系，以及输入和输出的变量，可以在编译时做静态优化，效率更高。在ES6中，使用import关键字引入模块，通过export关键字导出模块，但由于ES6目前的浏览器支持度原因，所以只能通过babel将不被支持的import编译为当前受到广泛支持的require。

注意：import是静态执行，所以不能使用表达式和变量等运行时才能得到结果的语法结构，所以无法实现条件加载，
命令只能在模块的顶层，不能在代码块之中；
import命令输入的变量都是只读的，因为它的本质是输入接口。也就是说，不允许在加载模块的脚本里面，改写接口，但是如果输出的是对象，允许修改对象属性；
import命令具有提升效果，会提升到整个模块的头部，首先执行；

JS中的原型对象

构造函数：为了解决从原型对象生成实例的问题，JavaScript提供了一个构造函数模式。构造函数其实是一个普通函数，但内部使用了this变量，对构造函数使用new运算符，可以生成实例对象，并且this变量会绑定在实例对象上。
JavaScript规定，每一个构造函数都有一个prototype属性，指向另一个对象（原型对象），这个对象的所有属性和方法，都会被构造函数的实例继承。

构造函数的继承

  1. 将子构造函数的prototype属性指向一个父构造函数的实例

注意：为了避免继承链的紊乱，在修改了prototype属性指向的原型对象后，须手动为新的原型对象加上constructor属性，指回原来的构造函数。

  1. 直接继承父构造函数的prototype

注意：与前一种方法相比，这样做的优点是效率比较高（不用执行和建立Animal的实例），比较省内存。缺点是 Cat.prototype和Animal.prototype现在指向了同一个对象，那么任何对Cat.prototype的修改，都会反应到Animal.prototype。

  1. 利用空对象作为中介

F是空对象，所以几乎不占内存，同时，修改Cat的prototype对象，不会影响到Animal的prototype对象。

  1. 拷贝继承，将父构造函数的prototype对象中的属性，拷贝给子构造函数的prototype对象。

  2. 构造函数绑定，使用call或apply方法，将父对象的构造函数绑定在子对象上。 

    function Cat(){
    Animal.apply(this, arguments);
}

    非构造函数的继承

  3. 浅拷贝 

    function extendCopy(p) {
    var c = {};
    for (var i in p) { 
        c[i] = p[i];
    }
    return c;
}

    把父对象的属性，全部拷贝给子对象。但是需要注意，如果父对象的属性为数组或对象，那么实际上，子对象获得的只是一个内存地址，而不是真正拷贝，因此存在父对象被修改的可能性。

  4. 深拷贝，通过递归调用浅拷贝实现真正意义上数组和对象的拷贝。 

    function deepCopy(p, c) {
    var c = c || {};
    for (var i in p) {
        if (typeof p[i] === 'object') {
            c[i] = (p[i].constructor === Array) ? [] : {};
            deepCopy(p[i], c[i]);
        } else {
            c[i] = p[i];
        }
    }
    return c;
}

  • 原型对象拥有constructor属性，指向构造函数；
  • 原型对象的用途是包含所有实例对象共有的属性和方法；

图片格式

jpg/jpeg：有损压缩、体积小、加载快、不支持透明。
png：无损压缩、质量高、体积大、支持透明。
gif：无损压缩、用于显示动态图片，不支持Alpha透明通道。
svg：文本文件、体积小、不失真、兼容性好，SVG是一种基于XML语法的图像格式。
bmp：图像信息丰富，无损，几乎不对数据进行压缩，因而占据空间较大，支持索引色和直接色。
webp：适用细节丰富的图片，支持透明，可以显示动态图片，集多种图片文件格式的优点于一身，但兼容性不好。

base64编码

文本文件、依赖编码、一种编码方式，和雪碧图一样，可作为小图标解决方案。

前端性能优化

前端性能优化

浏览器缓存机制

浏览器缓存机制有四个方面，按照获取资源时请求的优先级排列如下：

  • Memory Cache
  • Service Worker Cache
  • HTTP Cache
  • Push Cache

Memory Cache：内存中的缓存，优先级最高，是浏览器最先尝试去命中的一种缓存，响应速度最快。但由于内存资源有限，一般只有小文件会被放进Memory Cache，而大文件放进磁盘。
Service Worker Cache：Service Worker是一种独立于主线程之外的JS线程，脱离于浏览器窗体，无法直接访问DOM，一般用来实现离线缓存、消息推送和网络代理等功能。
HTTP Cache：分为强缓存和协商缓存两种，强缓存优先级较高，在命中强缓存失败的情况下，会走协商缓存。

  • 强缓存：利用HTTP头中的Expires和Cache-Control两个字段控制。在强缓存中，当请求发出时，浏览器会根据这两个字段去判断目标资源是否命中，若命中则直接从缓存中获取资源，不与服务端进行通信。
    Expires 是 HTTP/1.0 的产物，表示资源会在 Wed, 22 Oct 2018 08:41:00 GMT 后过期，需要再次请求。并且 Expires 受限于本地时间，如果修改了本地时间，可能会造成缓存失效。
    Cache-Control 出现于 HTTP/1.1，优先级高于 Expires 。该属性表示资源会在多少秒后过期，需要再次请求。 

Expires: XXX GMT（HTTP/1.0）
Cache-Control: max-age=XXX（HTTP/1.1）
  • 协商缓存：协商缓存依赖于服务端和浏览器之间的通信，浏览器需要向服务器询问缓存的相关信息，进而判断是否需要重新发起请求。如果服务端提示缓存资源未改动，资源会被重定向到浏览器缓存，对应状态码返回304。
    协商缓存有两种实现方式：Last-Modified 和 If-Modified-Since、ETag 和 If-None-Match。

Last-Modified和If-Modified-Since：Last-Modified 表示本地文件最后修改日期，If-Modified-Since 会将 Last-Modified 的值发送给服务器，询问服务器在该日期后资源是否有更新，有更新的话就会将新的资源发送回来。但是如果在本地打开缓存文件，就会造成 Last-Modified 被修改，所以在 HTTP / 1.1 出现了 ETag 。
ETag和If-None-Match：ETag类似于文件指纹，If-None-Match 会将当前ETag发送给服务器，询问该资源ETag是否变动，有变动的话就将新的资源发送回来，并且ETag优先级比Last-Modified高。
Push Cache：Push Cache是指HTTP2在server push阶段存在的缓存，是缓存的最后一道防线，当前三种缓存均未命中的情况下使用。Push Cache存在于会话阶段，当session终止时，缓存也随之释放。不同的页面只要共享了同一个HTTP2连接，就可以共享同一个Push Cache。

CDN

CDN有两个核心概念：缓存和回调。
缓存指把资源复制一份到CDN服务器上的这个过程，回调指CDN发现自己没有这个资源或资源失效，去向根服务器请求资源的过程。
使用：静态资源尽量使用CDN加载，由于浏览器对于单个域名有并发请求上限，可以考虑使用多个CDN域名。对于CDN加载静态资源需要注意CDN域名要与主站不同，否则每次请求都会带上主站的Cookie。

图片优化

图片大小

  • 减少像素点；
  • 减少每个像素点能够显示的颜色；

    图片加载

  • 使用CSS代替部分修饰类图片；
  • 计算适配屏幕的宽度，请求相应裁剪好的图片；
  • 小图使用base64格式；
  • 将多个图标文件整合到一张图片中（雪碧图）；
  • 选择正确的图片格式；

    渲染优化

    浏览器的渲染机制

  • 处理HTML并构建DOM树；
  • 处理CSS并构建CSSOM树；
  • 将DOM与CSSOM合并成一个渲染树；
  • 根据渲染树布局，计算每个节点的位置；
  • 调用GPU绘制，合成图层并显示；

预加载：预先获取资源，预加载是声明式的fetch，强制浏览器请求资源，不会阻塞onload事件。
预渲染：通过预渲染将下载的文件预先在后台渲染，预渲染虽然可以提高页面的加载速度，但是要确保该页面会被用户在之后打开，否则就会产生资源浪费。
懒执行：懒执行就是将某些逻辑延迟到使用时再计算。该技术可以用于首屏优化，对于某些耗时逻辑并不需要在首屏就使用的，就可以使用懒执行。懒执行需要唤醒，一般可以通过定时器或者事件的调用来唤醒。
懒加载：懒加载就是将不关键的资源延后加载。懒加载的原理就是只加载自定义区域内需要加载的东西。对于图片来说，先设置图片标签的src属性为一张占位图，将真实的图片资源放入一个自定义属性中，当进入自定义区域时，就将自定义属性替换为src属性，这样图片就会去下载资源，实现了图片懒加载。

回流与重绘

回流：对DOM的修改引发了DOM几何尺寸的变化时，浏览器需要重新计算元素的几何属性，再将计算结果绘制出来，这个过程称为回流。
重绘：对DOM的修改导致样式的变化，但并未影响几何属性时，浏览器不需要重新计算元素的几何属性，直接为元素绘制新的样式即可，这个过程称为重绘。
回流必定会发生重绘，重绘不一定会引发回流。回流所需的成本比重绘高的多，改变深层次的节点很可能导致父节点的一系列回流。在实际开发过程中，应尽可能减少回流与重绘。

HTTP（超文本传输协议）

HTTP协议的主要特点

  1. 简单快速：客户端向服务器请求服务时，只需传送请求方法和路径。由于HTTP协议简单，使得HTTP服务器的程序规模小，因而通信速度很快。
  2. 灵活：HTTP允许传输任意类型的数据对象，传输的类型由Content-Type加以标记。
  3. 无连接：HTTP使用了TCP连接，通信的双方在交换HTTP报文之前不需要先建立HTTP连接。
  4. 无状态：对于同一个客户第二次访问同一个服务器上的页面时，服务器的响应与第一次被访问时的相同，服务器不保留与客户端连接时的任何状态，这种无状态特性简化了服务器的设计。

HTTP协议的工作流程

  1. 客户端与服务器建立连接；
  2. 客户端向服务器发送请求；
  3. 服务器接接收请求，返回相应的响应报文；
  4. 客户端与服务器关闭连接；

HTTP的URL

  • 对于万维网的网点的访问要使用HTTP协议，通过URL去定位资源的位置。
  • HTTP的URL一般形式是：http://<主机>:<端口>/<路径>
  • HTTP的默认端口号是80，通常可以省略，URL中的字母不区分大小写。

HTTP报文结构

  HTTP有两类报文，请求报文和响应报文，都是由三个部分组成，区别在于开始行不同。

开始行：用于区分是请求报文还是响应报文。在请求报文中的开始行叫做请求行，而在响应报文中的开始行叫做状态行。开始行由三个字段组成，之间以空格分隔开。
首部行：用来说明浏览器、服务器或报文主体的一些信息。整个首部行结束时，还有一个空行将首部行和后面的实体主体分开。
实体主体：一般不使用此字段。

请求报文：请求报文的请求行有三个内容，即方法、请求资源的URL和HTTP的版本。
请求方法：

  • GET：请求读取由URL所标志的信息；
  • POST：给服务器添加信息；
  • HEAD：请求读取由URL所标志的信息的首部；
  • PUT：在指明的URL下存储一个稳定；
  • DELETE：删除指明的URL所标志的资源；
  • TRACE：用来进行环回测试的请求报文；
  • CONNECT：用于代理服务器；
  • OPTION：请求一些选项的信息；
    一个完整的HTTP请求报文示例：
    GET /dir/index.html HTTP/1.1  开始行：使用了相对URL
    Host: www.xyz.edu.cn  （首部行开始）给出了主机的域名
    Connection: close  告诉浏览器发送完请求的文档后就可以释放连接
    User-Agent: Mozilla/5.0  表明用户代理使用的是 Firefox
    Accept-Language: cn  表示用户希望优先得到中文版本的文档（首部行结束）（空行）

响应报文：响应报文的状态行有三个内容，即HTTP版本、状态码以及解释状态码的简单短语。
状态码：状态码都是三位数的，分为5大类，分别以数字1~5开头。

  • 1xx：表示通知信息，如收到请求或正在进行处理。
  • 2xx：表示成功，如接受或知道了。
  • 3xx：表示重定向，如要完成请求须采取进一步的行动。
  • 4xx：表示客户端出错，如请求中有错误的语法或不能完成。
  • 5xx：表示服务器差错，如服务器失效无法完成请求。

一个响应报文示例：
HTTP/1.1 202 Accepted

HTTP持久连接（从HTTP1.1开始支持）

非持久连接：HTTP协议采用“请求–应答”模式，当使用普通模式，即非Keep-Alive（持久连接）模式时，每个请求/应答客户端和服务器都要新建一个连接，完成之后立即断开连接（HTTP协议为无连接的协议）；
持久连接：当使用Keep-Alive模式（持久连接、连接重用）时，Keep-Alive功能使客户端到服务器端的连接持续有效，当出现对服务器的后继请求时，Keep-Alive 功能避免了重新建立连接。

Get与Post区别

Get和Post是Http协议中的两种发送请求的方法，而Http是基于TCP/IP的关于数据如何在万维网中通信的协议，Get和Post本质上就是TCP连接，并无差别。

  1. Get在浏览器回退时是无害的，而Post会再次提交请求；
  2. Get产生的URL地址可以被Bookmark，而Post不可以；
  3. Get请求会被浏览器主动cache，而Post不会；
  4. Get请求只能进行url编码，而Post支持多种编码方式；
  5. Get请求参数会被完整保留在浏览器历史记录中，而Post中的参数不会被保留；
  6. Get请求在URL中传送的参数是有长度限制的，而Post没有；
  7. 对参数的数据类型，Get只接受ASCII字符，而Post没有限制；
  8. Get比Post更不安全，参数直接暴露在URL上，不能用来传递敏感信息；
  9. Get参数通过URL传递，Post放在Request body中；
  10. Get产生一个TCP数据包，Post产生两个TCP数据包。对于Get方式的请求，浏览器会把http header和data一并发送出去，服务器响应200（返回数据）；而对于Post，浏览器先发送header，服务器响应100 continue，浏览器再发送data，服务器响应200 ok（返回数据）；

HTTP请求头内容

  1. 使用的HTTP动作：post/get；
  2. 浏览器所能接受的Content-type：Accept；
  3. 浏览器所能接受的语言和编码信息Accept-language，Accept-Encoding；
  4. 本机信息：浏览器类型、操作系统类型User-Agent；
  5. 请求的主机和端口Host；
  6. 使用的连接方式Connnection；
  7. Http/1.1 ，1.1为协议的版本，使用持久连接，1.0使用非持久连接；
  8. Cookie；

this浅析

什么是this？

  this是 JavaScript 语言的一个关键字，它是函数运行时，在函数体内部自动生成的一个对象，只能在函数体内部使用，指代函数当前的运行环境，实际上是一个指针，指向调用函数的对象。

this的绑定规则有哪些？

  1. 默认绑定；
  2. 隐式绑定；
  3. 显示绑定；
  4. new绑定；

this绑定规则详解：

默认绑定：在没有运用其他绑定规则时，即为默认绑定，非严格模式下，this指向全局对象，严格模式下，this指向undefined，undefined上没有this对象，会抛出错误。
隐式绑定：绑定到某个对象上，形式为xxx.fun()。
注意：若采用对象属性链的方式调用函数，只有最后一层会影响到调用位置，隐式绑定容易发生丢失的情况。
显示绑定：通过call, apply, bind的方式，显示地指定this所指向的对象。
new绑定：会创建一个新对象，并将this指向这个新的对象。

绑定优先级：

  当同时应用了多种绑定规则时，按以下优先级应用:new绑定>显示绑定>隐式绑定>默认绑定。

其他情况：

  1. 当将null或者是undefined作为this的绑定对象传入call、apply或者bind，这些值在调用时会被忽略，实际应用的是默认绑定规则。
  2. 箭头函数没有this，因此无法用call()、apply()、bind()这些方法去改变this的指向，它的this继承于外层代码库中的this。
